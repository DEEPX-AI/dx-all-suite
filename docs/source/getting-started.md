## 🧩 DX-Compiler: AI Model Compilation Scripts Guide

This section explains the purpose and flow of each script from `compiler-0_install_dx-compiler.sh` to `compiler-4_model_compile.sh`.

**🔄 Execution Order**

```bash
./getting-started/compiler-0_install_dx-compiler.sh
./getting-started/compiler-1_download_onnx.sh
./getting-started/compiler-2_setup_calibration_dataset.sh
./getting-started/compiler-3_setup_output_path.sh
./getting-started/compiler-4_model_compile.sh
```

**💡 Tip**

- The `.dxnn` file is the final executable generated by `dxcom`.
- Each script can be run independently, but following the order above ensures the entire process works as intended.

---

### 📁 0. compiler-0_install_dx-compiler.sh

Installs the dx-compiler package.

- **Purpose**: Install dx-compiler
- **Description**:
  - Executes `./dx-compiler/install.sh` to install the dx-compiler package.
  - **Automatically skips installation if `dxcom` is already installed** (unless `-f` or `--force` is used).

#### 📌 Key Function

- Navigates to the `dx-compiler` directory and runs the installation script.
- Checks if `dxcom` is already installed using `dxcom -v` with python virtual environment activation.
  - If `dxcom` is found, skips installation (unless `-f` or `--force` is specified).
  - Use `-f` or `--force` to reinstall even when `dxcom` is already present.
- Validates the installation exit code and reports success or failure.

#### 📌 Usage

```bash
# Install dx-compiler (default)
bash compiler-0_install_dx-compiler.sh

# Force reinstallation even if dxcom is already installed
bash compiler-0_install_dx-compiler.sh --force
# or
bash compiler-0_install_dx-compiler.sh -f
```

---

### 📁 1. compiler-1_download_onnx.sh

Downloads model files (`.onnx`, `.json`) and links them into the workspace.

- **Purpose**: Automatic model download
- **Description**:
  - Downloads files into `modelzoo/onnx` and `modelzoo/json` directories.
  - Supports models: `YOLOV5S-1`, `YOLOV5S_Face-1`, `MobileNetV2-1`.
  - `--force` option allows overwriting existing files.

#### 📌 Key Functions

- `show_help([type], [message])`

  - Displays usage and exits when invalid options are passed.
  - Supports `--force`, `--help`.

- `download(model_name, ext_name)`
- `download(model_name, ext_name)`

  - Downloads resources based on the given model name and extension.
  - Calls `get_resource.sh` and saves to `modelzoo/{ext_name}/{model_name}.{ext_name}`.
  - Also creates a symbolic link to the workspace (`workspace/modelzoo/`).

- `main()`
  Iterates through model list and extension list, calling `download()`.

---

### 📁 2. compiler-2_setup_calibration_dataset.sh

Downloads the calibration dataset and modifies `.json` config files to use it.

- **Purpose**: Calibration dataset setup
- **Description**:
  - Downloads `calibration_dataset.tar.gz` and extracts it to `workspace/dataset`.
  - Creates a symlink in the workspace: `workspace/dataset/calibration_dataset` → `./calibration_dataset`.
  - Forcefully changes (hijacks) the `dataset_path` entry in `modelzoo/json/*.json` to `./calibration_dataset`.
  - `--force` option allows overwriting existing files.

#### 📌 Key Functions

- `download()`

  - Downloads the calibration dataset archive from the remote server.
  - Calls `get_resource.sh` to download and extract `dataset/calibration_dataset.tar.gz`.
  - Creates a symbolic link from `workspace/dataset/calibration_dataset` to the extracted dataset.

- `hijack_dataset_path(model_name)`

  - Forcefully changes the `"dataset_path"` value in `json/{model_name}.json` to `./calibration_dataset`.
  - Backs up the original file as `.bak` and modifies the value using the `sed` command.
  - Outputs the `diff` showing changes before and after.

- `main()`
  - Executes `download()` to get the calibration dataset.
  - Runs `hijack_dataset_path()` for each of the three models.

---

### 📁 3. compiler-3_setup_output_path.sh

Creates a symbolic link from `./dxnn` to the workspace output path.

- **Purpose**: Setup path for compiled model output
- **Description**:
  - In host mode: `./dxnn` → `workspace/dxnn`
  - In container mode: `./dxnn` → `${DOCKER_VOLUME_PATH}/dxnn`
  - Automatically detects environment and handles broken symlinks

#### 📌 Key Function

- `setup_compiled_model_path()`
  - Checks if the environment is a container and determines the output path accordingly:
    - Container: `${DOCKER_VOLUME_PATH}/dxnn`
    - Host: `${DX_AS_PATH}/workspace/dxnn`
  - Creates a symbolic link `./dxnn` pointing to the appropriate workspace directory.
  - Handles recreation if the existing link is broken.

---

### 📁 4. compiler-4_model_compile.sh

Compiles `.onnx` files into `.dxnn` format using `dx_com`.

- **Purpose**: Run model compilation
- **Description**:
  - Uses `dx_com` to compile `onnx + json → dxnn`
  - Outputs result to `./dxnn/`

#### 📌 Key Functions

- `compile(model_name)`

  - Runs `dx_com` to convert `.onnx + .json` into a `.dxnn` file.
  - The output is saved in the `./dxnn` directory.
  - Exits on failure.

- `main()`
  - Iterates through the list of models and calls `compile()` for each.

## 🧩 DX-Runtime: Application Execution Scripts Guide

This document explains the role and execution flow of the scripts from `runtime-0_install_dx-runtime.sh` to `runtime-3_run_example_using_dxrt.sh`.
It serves as a guide for running compiled `.dxnn` models from the `dx-compiler` in an actual runtime environment based on example executions.

**🔄 Runtime Execution Order**

```bash
bash runtime-0_install_dx-runtime.sh
bash runtime-1_setup_input_path.sh
bash runtime-2_setup_assets.sh
bash runtime-3_run_example_using_dxrt.sh
```

**💡 Tip**

- Run `runtime-*` scripts **after** `.dxnn` models have been compiled successfully.
- `fim` is a CLI tool for viewing image output. It will be automatically installed if missing.
- Before running examples, make sure to run `dx_app/setup.sh` to download required models and sample data.

---

### 📁 0. runtime-0_install_dx-runtime.sh

Installs the dx-runtime package.

- **Purpose**: Install dx-runtime
- **Description**:
  - Executes `./dx-runtime/install.sh --all` to install the dx-runtime package with all components.
  - Supports `--exclude-fw` flag to exclude firmware installation.
  - **Automatically skips installation if dx-runtime is already installed** (unless `-f` or `--force` is used).

#### 📌 Key Function

- Navigates to the `dx-runtime` directory and runs the installation script.
- By default, runs with `--all` flag to install all components.
- When `--exclude-fw` flag is provided, runs with `--exclude-fw` to skip firmware installation.
- Checks if dx-runtime is already installed using `dx-runtime/scripts/sanity_check.sh`.
  - If dx-runtime is found, skips installation (unless `-f` or `--force` is specified).
  - Use `-f` or `--force` to reinstall even when dx-runtime is already present.
- Validates the installation exit code and reports success or failure.

#### 📌 Usage

```bash
# Install with all components (default)
bash runtime-0_install_dx-runtime.sh

# Install without firmware
bash runtime-0_install_dx-runtime.sh --exclude-fw

# Force reinstallation even if dxcom is already installed
bash runtime-0_install_dx-runtime.sh --force
# or
bash runtime-0_install_dx-runtime.sh -f

# Force reinstallation without firmware
bash runtime-0_install_dx-runtime.sh --exclude-fw --force
# or
bash runtime-0_install_dx-runtime.sh --exclude-fw -f
```

---

### 📁 1. runtime-1_setup_input_path.sh

Creates a symbolic link to the compiled model directory for runtime access.

- **Function**: Set runtime model path
- **Description**:
  - Creates a symbolic link for `./dxnn` to point to `workspace/dxnn`.
  - Automatically detects and supports both host and Docker container environments.

#### 📌 Key Function

- `setup_compiled_model_path()`
  - Automatically sets the path based on whether the environment is a container or not.
    - Container: `${DOCKER_VOLUME_PATH}/dxnn`
    - Host: `${DX_AS_PATH}/workspace/dxnn`
  - Links `./dxnn` to the respective workspace path (also restores broken symlinks).

---

### 📁 2. runtime-2_setup_assets.sh

Prepares the configuration files and model resources for running examples.

- **Function**: Prepare configuration files and resources for running examples.
- **Description**:
  - Calls the `setup.sh` of `dx_app` and `dx_stream` to download and copy necessary resources for example execution.
  - Automatically prepares required models, configuration files, and sample images.

#### 📌 Key Function

- `setup_assets(target_path)`
  - Executes the `setup.sh` of each module (`dx_app`, `dx_stream`).
  - Internally copies or links sample images, JSON configurations, models, etc.

---

### 📁 3. runtime-3_run_example_using_dxrt.sh

Runs the `.dxnn` model using `dx_app` async examples and checks the results.

- **Function**: Run runtime examples (Object Detection, Face Detection, Image Classification).
- **Description**:
  - Forks the `dx_app` example binaries and sample data into the `forked_dx_app_example` folder.
  - Executes `yolov5face_async`, `yolov5_async`, and `efficientnet_async` binaries with compiled `.dxnn` models.
  - Automatically detects TTY/headless environments and adjusts display behavior accordingly.
  - Validates model files exist before execution.

#### 📌 Key Functions

- `fork_examples()`

  - Copies `dx_app/bin` binaries: `yolov5face_async`, `yolov5_async`, `efficientnet_async`.
  - Copies sample images for object detection, face detection, and classification.
  - Uses error handling with `handle_cmd_failure()` to provide helpful hints if binaries are missing.

- `run_example(exe_file_path, dxnn_file_path, image_path, save_log, loop)`

  - Executes the async binary with specified model and input image.
  - `loop`: Number of inference iterations (default: 300, face detection: 30).
  - `save_log`: Redirects output to `result-app.log` when set to "y".
  - Automatically adds `--no-display` flag in TTY/headless environments (checked via `/deepx/tty_flag`).
    - **Exception**: `efficientnet_async` does not support `--no-display` option, so the flag is not added for this binary.
  - Exits on failure with error message.

- `show_result(result_path)`

  - In graphical environments: Prompts user to continue (legacy `fim` viewer code disabled).
  - In TTY environments: Displays instructions for copying results using `docker cp`.
  - Handles both environments gracefully.

- `main()`
  
  - Validates all `.dxnn` model files exist (`YOLOV5S_Face-1`, `YOLOV5S-1`, `MobileNetV2-1`).
  - Checks and installs `fim` if needed (for future image viewing support).
  - Runs three examples sequentially:
    - **YOLOV5 Face**: 30 iterations on `face_sample.jpg`
    - **YOLOV5S**: 300 iterations on `1.jpg`
    - **MobileNetV2**: 300 iterations on `ILSVRC2012/1.jpeg` with logged output

**Models tested:**
- YOLOV5S_Face-1 (Face Detection)
- YOLOV5S-1 (Object Detection)
- MobileNetV2-1 (Image Classification)
